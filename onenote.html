实验一： 在工作区添加一个新的文件或者修改了代码，这时候执行git diff，会出现什么情况？ 如果都提交到了仓库，暂存区没有文件，那么git diff没有情况 实验二： 在工作区添加一个新的文件或者修改了代码，这时候，执行了git add .,再执行了git diff,会出现什么情况？ 有与暂存区比较的情况 实验三： 在工作区添加一个新的文件或者修改了代码，这时候，执行了git add .，这时候再修改了工作区中的代码，再执行了git diff,会出现什么情况？ 出现两种颜色，有之前和之后的比较 结论： git diff永远是在拿工作区做为基准， 情况一：如果暂存区是空的，则是拿工作区和仓库在进行比较 情况二：如果暂存区不是空的，则是拿工作区和暂存区进行比较 1、如果执行了git add,则工作区和暂存区没有任何区别，这时候会进行工作区和暂存区代码的比较，但是比较不出任何结果 2、如果执行了git add,同时随后又修改了工作区中的代码，这时候就会进行工作区和暂存区代码的比较，这时候比较是有结果的 实验四： 在工作区添加一个新的文件或者修改了代码，这时候，执行了git add .这时候假如我们执行git diff，是不会有任何结果的，但是我们执行git diff --cached,会出现什么情况？ 结论： 如果暂存区和仓库当中的代码提交比较，使用git diff --cached 1.如果把修改第一次的文件放到了缓存区，但是并没有放到仓库，这时候发现我想重新改一下，就可以直接再一次添加到暂存区，并且新添加的那一次会覆盖掉暂存区原来的文件。 2.如果是git diff 表示的是比较工作区与暂存区的不同 如果是git diff --cached 表示的是暂存区与上一次提交的不同 3.3.分支与分支之间是平等的，所以可以随便任意两条分支合并
